# Puppet Deployment Integration

This file describes how yaamon is deployed via Puppet and what the surrounding
infrastructure provides, so that features can be implemented to match.

## Deployment topology

```
Internet / Tailscale
        │
        ▼
  caddy (macvlan IP, ports 80 + 443)
  ├── TLS termination (Let's Encrypt cert)
  ├── HTTP → HTTPS redirect (301)
  ├── oauth2-proxy sidecar (forward auth, port 4180)
  │     └── OIDC provider: Kanidm
  └── reverse proxy → yaamon:80
        │
        ▼
  yaamon (bridge network, port 80, TLS disabled)
```

Caddy is the only container on the host's macvlan interface.  yaamon is on
the default Docker bridge network and is not directly reachable from outside.

## Container mounts

| Host path                              | Container path    | Mode |
|----------------------------------------|-------------------|------|
| `/vol/services/yaamon/yaamon/config/`  | `/etc/yaamon`     | ro   |
| `/vol/services/yaamon/yaamon/data/`    | `/var/lib/yaamon` | rw   |

All files in `/etc/yaamon` are managed by Puppet and are read-only inside
the container.  The app must not attempt to write to that directory.

## config.yaml (generated by Puppet)

```yaml
server:
  http_port: 80
  https_port: 443   # unused; caddy owns 443
  redirect_http: false

tls:
  mode: disabled    # caddy handles TLS
```

## Proxy authentication headers

Every request that reaches yaamon has already been authenticated by the
oauth2-proxy.  Caddy copies the following headers from the oauth2-proxy auth
response onto the upstream request before forwarding to yaamon:

| Header                          | Content                                      |
|---------------------------------|----------------------------------------------|
| `X-Auth-Request-User`           | OIDC subject / username                      |
| `X-Auth-Request-Preferred-Username` | Human-readable preferred username        |
| `X-Auth-Request-Email`          | User's email address                         |
| `X-Auth-Request-Groups`         | Comma-separated list of Kanidm group names   |

These headers are set by oauth2-proxy only when a valid session cookie is
present.  Requests without a valid session are redirected to the oauth2 login
page before they reach yaamon, so yaamon will never see a request with missing
or forged auth headers (assuming it is not reachable except through caddy).

### Example header values

```
X-Auth-Request-User: jch
X-Auth-Request-Preferred-Username: jch
X-Auth-Request-Email: jch@honig.net
X-Auth-Request-Groups: yaamon_admins,yaamon_users
```

Group names come directly from Kanidm and are configured by the Kanidm
administrator.  The Puppet module sets `oauth2_allowed_groups` in hiera to
control which groups are permitted to reach the app at all; additional
group-to-permission mapping is left to yaamon.

## Kanidm groups and claim values

Access is gated on membership in `yaamon_access`; users not in that group are
rejected by oauth2-proxy before reaching yaamon.

The Kanidm oauth2 application maps groups to values in the custom claim
`yaamon_group` (an array).  oauth2-proxy exposes this claim as the
`X-Auth-Request-Groups` header.  The mapping is:

| Kanidm group       | Claim value        | Intended role       |
|--------------------|--------------------|---------------------|
| `yaamon_access`    | `yaamon_access`    | readonly (default)  |
| `yaamon_rw`        | `yaamon_rw`        | read-write          |
| `yaamon_admin`     | `yaamon_admin`     | admin               |
| `yaamon_superadmin`| `yaamon_superadmin`| superuser           |

A user may be in more than one group; yaamon should grant the highest
role present in the header.

## Suggested config.yaml additions for proxy auth

```yaml
proxy_auth:
  # Which header carries the username.  Prefer preferred_username; fall back
  # to user if the OIDC provider does not populate preferred_username.
  username_header: X-Auth-Request-Preferred-Username

  # Which header carries the comma-separated group claim values.
  groups_header: X-Auth-Request-Groups

  # Map claim values to yaamon roles.
  # Grant the highest role present; deny if none match.
  group_permissions:
    yaamon_superadmin: superuser
    yaamon_admin:      admin
    yaamon_rw:         readwrite
    yaamon_access:     readonly

  # Auto-create a DB user on first OAuth login (password stored as "*";
  # local login is denied for these accounts).
  create_users: true

  # If true, update the DB user's role to the OAuth-mapped role on every
  # successful auth.  If false (default), the mapped role is used only for
  # the session and the DB record is not modified.
  update_db_role: false
```

## Tailscale authentication

caddy-tailscale injects these headers for requests arriving from tailnet
peers.  No Tailscale API or socket access is used — headers only.

| Header                      | Content                                        |
|-----------------------------|------------------------------------------------|
| `Tailscale-User-Login`      | Tailscale login name (e.g. `jch@honig.net`)   |
| `Tailscale-User-Name`       | Display name (may be absent)                   |
| `Tailscale-User-Profile-Pic`| Avatar URL (may be absent)                     |

### Suggested config.yaml additions for Tailscale auth

```yaml
tailscale_auth:
  enabled: false
  user_header:   Tailscale-User-Login
  name_header:   Tailscale-User-Name
  avatar_header: Tailscale-User-Profile-Pic
```

### Behaviour

1. If `tailscale_auth.enabled` and `Tailscale-User-Login` is non-empty,
   search DB users whose `tailscale_usernames` profile field contains that
   login.
2. **Match found** — establish a session using the DB user's ID and stored
   role.  If the DB `full_name` or `avatar_url` are empty, fill them from
   the Tailscale headers.
3. **No match** — fall through to cookie-based session check, then to the
   login prompt.  Users without a Tailscale mapping can still log in with a
   password.

### User profile field

Each DB user has a `tailscale_usernames` field (comma-separated Tailscale
login names, e.g. `jch@honig.net`).  Editable in the profile modal
(self-service) and in the admin user editor.

## Auth middleware architecture

Both Tailscale and OAuth2 auth are **stateless per-request** — the session is
re-derived from headers on every request and no session cookie is written.  A
`proxyAuthMiddleware` runs before the cookie session middleware.  All
downstream handlers call `auth.FromContext()` and are unaware of which auth
path was used.

Middleware order:

```
RealIP → Logger → Recoverer → CSRF → setupGuard
  → proxyAuthMiddleware   (Tailscale + OAuth2 header auth)
  → sessions.Middleware   (cookie auth; fires only if no proxy session set)
  → validateSessionUser
```

## OAuth2 user provisioning

### User not found in DB

- `create_users: true` (default) — create the user with the OAuth-mapped
  role.  Password stored as `*`; local login is denied for this account.
- `create_users: false` — return 403.

### User found in DB

- Session role = OAuth-mapped role (from groups header), not the DB value.
- `update_db_role: true` — also write the mapped role back to the DB record
  on each successful auth.
- `update_db_role: false` (default) — DB role is untouched; the OAuth role
  is used only for the current session.

## Dropdown UI — username and auth method indicator

The user dropdown shows the **username** above the role.  When the session
was established via OAuth2 or Tailscale, a shield icon (Bootstrap Icons
`bi-shield-check`) appears next to the username.  Hovering shows a tooltip:

- OAuth2: `Authenticated via OAuth2`
- Tailscale, login matches DB username: `Authenticated via Tailscale`
- Tailscale, login differs from DB username:
  `Authenticated via Tailscale (jch@honig.net)`

Two new `pageData` fields carry this information:

| Field             | Type   | Content                                              |
|-------------------|--------|------------------------------------------------------|
| `AuthMethod`      | string | `"oauth2"`, `"tailscale"`, or `""` (local)          |
| `ExternalUsername`| string | Tailscale login / OAuth subject when ≠ DB username   |


## Suggested behaviour

1. **Header present → proxy auth path**: If `X-Auth-Request-Preferred-Username`
   (or whichever header is configured) is non-empty, skip the password login
   flow entirely.  Derive the username and role from the headers and
   establish a session (or treat each request as stateless if preferred).

2. **Group mapping**: Walk `X-Auth-Request-Groups` (split on `,`).  Assign the
   highest role found in `group_permissions`.  If no group matches, return 403.

3. **No headers → local auth**: If the proxy auth headers are absent (e.g.
   direct access during development, or proxy auth not configured), fall back
   to the existing username/password login flow.

4. **Health endpoint exemption**: `/health` must remain unauthenticated so the
   Docker `HEALTHCHECK` and Caddy health checks continue to work.  The oauth2
   forward-auth rule in the Caddyfile already exempts `/oauth2/*`; the health
   endpoint is accessed directly by the Docker daemon on `localhost` and bypasses
   Caddy entirely.

5. **Settings file (`/etc/yaamon/settings.yaml`)**: Puppet still writes and
   applies this file at container startup via `YAAMON_STATE_FILE`.  It seeds
   the database with users and nodes.  Users seeded this way are for local
   auth fallback; proxy auth users need not exist in the DB (their session is
   derived entirely from headers).

## Hiera parameters (Puppet side)

These are the parameters the operator sets in node hiera to control the
oauth2-proxy configuration:

| Parameter                       | Purpose                                         |
|---------------------------------|-------------------------------------------------|
| `yaamon::oauth2_client_id`      | oauth2 application name in Kanidm (default: `yaamon`) |
| `yaamon::oauth2_client_secret`  | Client secret from Kanidm (goes in `duncan.eyaml`) |
| `yaamon::oauth2_allowed_groups` | Groups permitted to log in at all               |
| `yaamon::oauth2_set_xauthrequest` | Must be `true` to enable the auth headers     |
| `yaamon::tailscale_oauth_secret`| Tailscale OAuth key for the caddy-tailscale container |

## Development and testing workflow

Because these auth features require a fully-configured deployment (Caddy +
oauth2-proxy + Tailscale) to test end-to-end, feature branches are built as
Docker images and pushed to `ghcr.io` so they can be pulled into the real
instance.

### Branch image tags

A new GitHub Actions workflow (`.github/workflows/branch.yml`) triggers on
pushes to any branch **except `main`**:

1. Runs `goreleaser release --snapshot --clean --skip=docker` to build the
   binary and `.deb`.
2. Builds a Docker image tagged with the **branch name**:
   `ghcr.io/jchonig/yaamon:<branch-name>`
3. Also tags with the short SHA for pinning:
   `ghcr.io/jchonig/yaamon:<branch-name>-<sha>`

No GitHub Release or `latest` tag is created.  The `latest` tag is only
updated by the release workflow on a `v*` tag push.

### Testing against a branch image

In the production `docker-compose.yml`, temporarily change the image tag:

```yaml
image: ghcr.io/jchonig/yaamon:auth   # branch name
```

Restart the container and test.  Revert to `latest` when done.

### Branch naming convention

Use descriptive branch names — they become the image tag:

| Branch name | Image tag |
|-------------|-----------|
| `auth`      | `ghcr.io/jchonig/yaamon:auth` |
| `feature/base-path` | `ghcr.io/jchonig/yaamon:feature-base-path` (slashes → dashes) |

### CI file to add

`.github/workflows/branch.yml` — mirrors the `test-deb` job from
`release.yml` for the snapshot build, then adds a Docker build/push step
using `docker/build-push-action` with the branch-name tag.  Requires
`packages: write` permission (already set on the repo).
